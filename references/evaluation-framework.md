# 五维生产级评估体系

本文件定义了方案在输出给用户之前必须通过的五维评估标准。每个维度有明确的准入阈值，不达标必须整改后复评。

---

## 评估流程

```
[七维拆解完成] → [逐维度评估] → {全部达标} → [输出终稿]
                              → {有不达标} → [整改对应维度] → [复评] → [全部达标] → [输出终稿]
```

最多整改 2 轮。如果 2 轮后仍有不达标维度，在终稿中标注"该维度未达标"并说明原因和风险。

---

## 维度一：业务适配性

**评估目标**：方案是否真实解决用户提出的业务问题，而不是套模板。

### 检查项

| # | 检查内容 | 准入阈值 |
|---|----------|----------|
| 1 | 用户提到的每个业务痛点，在方案中是否有对应的解决动作 | 100% 覆盖 |
| 2 | 每个业务目标是否有可量化的验收标准 | 100% 有量化指标 |
| 3 | 方案的约束条件（预算/时间/范围）是否与用户声明的完全一致 | 零偏差 |
| 4 | 目标受众的特征是否体现在方案设计中 | 至少 2 处体现 |

### 常见不达标原因
- 方案中出现了用户没有提到的"额外功能"或"增值建议"（过度设计）
- 验收标准模糊，如"提升用户体验""优化流程"（不可量化）
- 预算超限或时间超期但未说明取舍逻辑

---

## 维度二：技术可行性

**评估目标**：方案中涉及的技术路径、工具、平台是否真实可用，不是空想。

### 检查项

| # | 检查内容 | 准入阈值 |
|---|----------|----------|
| 1 | 技能梳理中提到的每个工具/平台是否真实存在且适用于该场景 | 100% 真实可用 |
| 2 | WBS 中的技术类子任务是否有明确的执行方法 | 每个子任务有方法 |
| 3 | 方案中是否避免了已被淘汰或即将淘汰的技术 | 零淘汰技术 |
| 4 | 技术选型是否考虑了团队现有能力（如团队无技术背景则不应要求自研） | 匹配团队能力 |

### 常见不达标原因
- 技能梳理写了"AI/机器学习"但项目中实际用不到
- 要求没有技术背景的团队自建系统（应推荐 SaaS/外包方案）
- 子任务只写了"开发XX系统"没有说明用什么技术栈

---

## 维度三：性能稳定性

**评估目标**：方案执行后能否稳定运行，关键路径是否有兜底方案。

### 检查项

| # | 检查内容 | 准入阈值 |
|---|----------|----------|
| 1 | 关键路径上的每个环节是否有失败后的备选方案 | 关键路径 100% 有兜底 |
| 2 | 高风险任务（可行性评估中标记为"高风险"的）是否有具体应对策略 | 每个高风险有策略 |
| 3 | 方案中是否有明确的"停止条件"（什么情况下项目应该暂停或终止） | 有停止条件 |
| 4 | 预算中是否有机动预备金 | 预备金 7%-15% |

### 常见不达标原因
- 施工类项目的关键材料没有备用供应商
- IT 项目没有回滚方案
- 没有说明什么情况下项目应该终止（如预算超支 20% 以上）

---

## 维度四：安全合规性

**评估目标**：方案是否符合行业法规和安全标准，行业红线是否全部覆盖。

### 检查项

| # | 检查内容 | 准入阈值 |
|---|----------|----------|
| 1 | 行业红线是否全部识别并纳入流程 | 行业红线 100% 覆盖 |
| 2 | 需要审批/许可的环节是否已纳入 WBS 和流程图 | 审批节点已纳入 |
| 3 | 数据安全/隐私保护要求是否已体现（如涉及用户数据） | 涉及数据的场景有保护措施 |
| 4 | 方案中是否避免了明显的安全隐患 | 零安全隐患 |

### 行业合规必检项

| 行业 | 必检合规项 |
|------|-----------|
| 餐饮/零售 | 食品安全认证、消防审批、门头城管审批、环保审批 |
| 电商/互联网 | 个人信息保护法、支付牌照、ICP备案、数据出境合规 |
| 制造业 | 安全生产许可、环评、设备第三方验收、ISO体系 |
| 教育/培训 | 办学许可、教师资格证、课程备案、资金监管账户 |
| 活动策划 | 公安报批（200人+）、消防验收、应急预案备案 |
| 房地产/建筑 | 规划许可、施工许可、竣工验收、环评能评 |

### 常见不达标原因
- 方案中完全未提及行业合规要求
- 涉及用户数据但未提及隐私保护
- 施工类项目未考虑消防审批周期

---

## 维度五：可扩展性

**评估目标**：方案是否具备后续迭代空间，不会因为本次实施导致技术/架构锁定。

### 检查项

| # | 检查内容 | 准入阈值 |
|---|----------|----------|
| 1 | 核心架构是否支持增量扩展（而非推倒重来） | 支持增量 |
| 2 | 供应商/技术选型是否避免了单一锁定 | 核心环节有替代选项 |
| 3 | 预算规划是否为后续迭代预留了空间 | 预备金 + 首期方案可独立运行 |
| 4 | 交付物是否具备复用价值（如VI手册、SOP文档） | 至少 2 个可复用交付物 |

### 常见不达标原因
- 技术选型绑定单一供应商且无替代方案
- 空间设计完全定制化，后续修改需全部重做
- 交付物是一次性的（如只做活动执行不做沉淀）

---

## 评估评分标准

每个维度的检查项逐项评估：

| 结果 | 含义 |
|------|------|
| **达标** | 所有检查项全部满足准入阈值 |
| **不达标** | 有 1 个及以上检查项未满足准入阈值 |

### 综合判定

- **全部达标**（5/5）：方案通过，输出终稿
- **需整改后复评**（有不达标维度）：执行整改→复评→直到全部达标

---

## 整改流程

```
[发现不达标维度]
    │
    ├── 定位问题：确认是哪个检查项不达标
    ├── 回溯到七维拆解中对应的维度
    ├── 修改方案内容（不是修改评估标准）
    ├── 重新评估该维度
    └── 如果达标→进入下一个不达标维度；如果仍不达标→标记为"未达标"并在终稿中说明
```

### 整改原则

1. **改方案不改标准**：评估标准不可降低，只能改方案内容来满足标准
2. **最小改动**：只改与不达标检查项直接相关的部分，不做无关修改
3. **不改约束**：不通过放松用户约束条件来"达标"（如不能通过增加预算来达标）
4. **记录轨迹**：每次整改需记录"改了什么→为什么改→改后结果"

## 回归对比与基线管理 (L3.5 Regression)

本模块定义执行结果与历史基线（baseline）的对比方法论，用于检测质量退化或进步。

### 基线文件

基线数据存储在 `<skill-base>/evals/regression-baseline.json`，包含：

- **test_results**：每个测试用例的评分、维度分、findings、misses
- **aggregate**：总体均值、标准差、各维度均值

### 对比流程

```
[当前执行完成] → [加载 baseline] → [匹配同行业条目] → [计算 Regression Delta] → [输出对比报告]
```

### Regression Delta 计算

| 指标 | 公式 | 解读 |
| :--- | :--- | :--- |
| 总分 Delta | `当前总分 - baseline 总分` | +X 进步 / -X 退化 |
| 维度 Delta | `当前维度分 - baseline 维度分` | 逐维度细粒度对比 |
| Miss 修复率 | `已修复 miss 数 ÷ baseline misses 总数` | 历史问题解决比例 |
| 新增 Miss 率 | `新增 miss 数 ÷ 当前总期望数` | 新引入的问题比例 |

### 质量退化阈值

| 预警级别 | 触发条件 | 响应动作 |
| :--- | :--- | :--- |
| **观察级** | 任一维度 Delta < -0.2 | 在知识库中记录退化观察 |
| **预警级** | 任一维度 Delta < -0.5 | 输出 **[质量退化预警]**，建议回溯原因 |
| **阻断级** | 任一维度 Delta < -1.0 | 标记当前方案为"需质量审查"，建议人工介入 |

### 基线更新策略

| 场景 | 更新动作 |
| :--- | :--- |
| 当前得分 > baseline 同行业得分 | 更新 baseline 为新得分（进步即新基线） |
| 当前得分 < baseline 但无退化预警 | 不更新（维持高质量基线） |
| 退化预警触发但已复盘修正 | 保留旧基线，追加新记录作为参照 |

### 审计维度定义 (L3 Auditor)

| 维度 | 测量方法 | 理想值 |
| :--- | :--- | :--- |
| **覆盖率** | 用户需求点数 ÷ 有对应方案动作的需求点数 | ≥ 90% |
| **量化率** | 验收标准中可量化的指标数 ÷ 总验收指标数 | ≥ 80% |
| **风险密度** | 风险预警标记数 ÷ 方案正文段落数 | 0.15 - 0.30 |
| **行业贴合度** | 引用的行业 SOP 是否符合用户行业 | 匹配 |
| **KPI 完整性** | 五维评估中每个维度是否都有评分 | 5/5 维度有评分 |

### 优化建议分类 (L4 Optimizer)

| 优化类型 | 触发条件 | 优先级 |
| :--- | :--- | :--- |
| 覆盖补全 | 覆盖率 < 90% | P0 |
| 量化增强 | 量化率 < 80% | P0 |
| 行业加厚 | 行业贴合度 = 部分匹配 | P1 |
| 遗漏修复 | 历史 miss 重现 | P1 |
| 密度调整 | 风险密度 < 0.15 或 > 0.30 | P2 |
