# 参与方界定与防甩锅手册

本文档明确全流程参与方的权责边界，列出常见推诿场景及防范方案，保护甲方免受乙方或第三方误导。

---

## 1. 角色权责边界表

| 角色类型 | 核心权责 | 甲方需警惕的风险点 |
| :--- | :--- | :--- |
| **甲方 (Demand)** | 明确业务目标、提供决策、支付费用、验收成果 | 需求朝令夕改导致乙方索赔；决策流程过长拖慢进度 |
| **乙方 (Vendor)** | 方案设计、技术实现、项目执行、售后支持 | 用实习生替换投标时的资深专家；低价中标后通过增项收费 |
| **第三方 (3rd Party)** | 提供特定组件（云、API）、监理、外审 | 接口文档不全；故障时与乙方互相踢皮球 |
| **监管方 (Regulator)** | 合规审批、资质核查、安全检查 | 政策突变导致项目停摆；现场检查不合格要求返工 |
| **内部协同方 (Internal)** | 配合数据对接、提供场地、业务试运行 | 部门利益冲突导致配合度低；信息传递断档 |

---

## 2. 常见“推诿/甩锅”场景与防范方案

### 场景 A：接口/数据对接失败
- **推诿词**：“我们的接口没问题，是对方的数据格式不标准/环境不对。”
- **防范方案**：
  1. 要求双方在项目启动阶段签署《接口规范协议》。
  2. 建立三方联调日志记录，明确标注每条请求的发出方、接收方及响应时间。
  3. 引入独立的测试账号，交叉验证各方接口的可用性。

### 场景 B：进度延期
- **推诿词**：“是因为甲方需求变动太频繁，且资料提供不及时。”
- **防范方案**：
  1. 建立《需求变更审批表》，明确记录变动点、预计工期影响及额外成本。
  2. 使用共享的任务看板，标注每项任务的“前置依赖”和“当前阻塞人”。
  3. 在合同中约定“甲方确认时限”和“资料交付清单”。

### 场景 C：效果不达标
- **推诿词**：“虽然数据没达到预期，但我们已经按合同要求完成了所有功能开发。”
- **防范方案**：
  1. 将“业务效果指标”与“功能实现指标”拆分，并分别约定付款权重。
  2. 验收标准中明确“高并发场景”、“极端数据情况”下的表现。
  3. 引入第三方监理或专业测试机构进行独立验收。

### 场景 D：隐性收费/增项
- **推诿词**：“这部分属于合同范围外的特殊需求，必须额外收费。”
- **防范方案**：
  1. 在招标阶段提供详尽的《需求说明书（PRD）》，要求乙方签字确认已充分理解。
  2. 约定“总价包干”或“变更金额上限（如不超过总额 5%）”。
  3. 要求乙方在报价单中列出所有可能的“可选增项”及其单价。

---

## 3. 甲方防范 Checklist

- [ ] 是否已明确项目唯一对接人（SPOC）？
- [ ] 关键沟通记录（微信、会议）是否已通过邮件或共享文档沉淀？
- [ ] 是否已建立“风险预警机制”，进度偏差超过 10% 必须书面汇报？
- [ ] 乙方参与项目的核心成员名单是否已写入合同（禁止私自更换）？
- [ ] 是否已明确第三方组件的费用承担方（含续费）？
