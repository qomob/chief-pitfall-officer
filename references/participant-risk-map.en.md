# Participant Risk Map & Anti-Buck-Passing (EN)

This document defines responsibility boundaries for participants and provides prevention plans for common "buck-passing" scenarios.

---

## 1. Responsibility Matrix

| Role | Core Responsibility | Key Risks for Client |
| :--- | :--- | :--- |
| **Client** | Goals, Decisions, Payments, Acceptance | Changing needs frequently; slow decision cycles |
| **Vendor** | Design, Execution, Support | Using junior staff instead of experts; hidden upsells |
| **3rd Party** | APIs, Cloud, Audit, Supervision | Poor docs; kicking the ball between vendor and 3rd party |
| **Regulator** | Compliance, Safety, Permits | Policy shifts; onsite inspection failures |

---

## 2. Common "Buck-Passing" Scenarios

### Scenario A: Interface/Data Integration Failure
- **Excuse**: "Our API is fine; their data format is non-standard."
- **Prevention**:
  1. Sign a "Standard Interface Agreement" at kickoff.
  2. Maintain a 3-party debug log with timestamps.

### Scenario B: Schedule Delays
- **Excuse**: "The client changed requirements and didn't provide materials on time."
- **Prevention**:
  1. Use a "Change Request Form" with impact analysis.
  2. Use a shared Kanban with "Prerequisites" and "Blockers" labeled.

---

## 3. Client Protection Checklist

- [ ] Clear Single Point of Contact (SPOC) defined?
- [ ] Critical communication (WeChat/Meeting) recorded in shared docs?
- [ ] "Risk Warning Mechanism" in place for delays > 10%?
- [ ] Core vendor team names written into the contract (no unauthorized swaps)?
