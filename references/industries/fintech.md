# 金融科技/数字银行适配指南

### 典型业务场景
- 银行/证券核心系统信创改造
- 普惠金融风控模型搭建
- 移动支付与聚合支付方案
- 金融数据中台与治理项目

### 环境核查要点
- **政策**：信创（信息技术应用创新）比例要求、央行合规审批。
- **技术**：金融级一致性（ACID）测试、高并发压力测试。

### 参与方风险预警
- **角色**：外包开发。**风险**：代码中混入后门或逻辑漏洞，引发资金风险。
- **角色**：三方风控。**风险**：模型数据过时，导致坏账率激增。
- **防甩锅方案**：强制执行代码审计（代码审计报告）；约定数据模型的动态更新频率。

### 验收关键项
- **安全性**：第三方渗透测试报告（零高危漏洞）。
- **性能**：核心交易响应时间 ≤ 100ms。
- **人工**：核心研发人员的背景背调（金融从业禁入名单核对）。

### 落地执行蓝图 (金融科技)

#### 1. 前期需求调研清单 (甲方自查)
- [ ] **监管合规**：确认项目是否需要央行/银保监会的审批或备案，信创改造比例是否达标。
- [ ] **数据安全**：确认数据分级分类（公开/内部/敏感/机密），跨境数据是否有出境审批需求。
- [ ] **系统现状**：梳理核心系统架构（集中式/分布式）、数据库类型、与外围系统的接口清单。
- [ ] **灾备能力**：确认现有灾备等级（同城双活/异地灾备），RPO/RTO 指标是否满足监管要求。

#### 2. 服务商评标标准
- **资质权重 (30%)**：金融行业安全服务资质、信创产品认证、等保三级/四级测评资质。
- **经验权重 (30%)**：近 2 年有 3 个以上同类型金融机构（银行/证券/支付）落地案例。
- **安全审计 (20%)**：是否接受甲方指定的第三方代码审计及渗透测试。
- **售后响应 (20%)**：S1 级故障 30 分钟内响应，2 小时内提供临时方案。

#### 3. 执行节点与巡检计划
- **架构评审阶段 (★核心)**：乙方技术方案必须通过甲方架构评审委员会审核，重点审查数据一致性、高并发、灾备切换方案。
- **代码审计阶段 (★核心)**：开发完成后引入第三方代码审计，重点排查后门、逻辑漏洞、硬编码密钥。
- **压力测试阶段**：模拟峰值流量（如双十一/季末结算），验证系统在极限负载下的表现。
- **灰度上线阶段**：先在 5%-10% 流量上灰度验证，确认无误后逐步放量至 100%。

#### 4. 合规资质办理指引
- **信创认证**：确认软硬件产品是否在信创目录内，是否符合国产化替代比例要求。
- **等保测评**：金融系统需通过等保三级或四级测评，测评周期 2-3 个月。
- **数据出境安全评估**：涉及跨境数据的，需向网信办申报数据出境安全评估。
- **金融业务许可**：如涉及支付/信贷业务，确认是否需要相应牌照或备案。

#### 5. 应急预案 (Plan B)
- **风险 1：上线后出现资金安全漏洞** → 触发阈值：任何涉及资金计算的错误。补救：立即回滚至上一稳定版本，乙方全额承担资金损失，甲方有权单方面终止合同。
- **风险 2：性能不达标** → 触发阈值：核心交易响应时间 > 200ms 或并发处理能力低于约定值的 80%。补救：乙方免费进行性能调优，调优期限不超过 2 周，逾期按违约处理。
- **风险 3：信创兼容性问题** → 触发阈值：核心功能在信创环境下无法正常运行。补救：乙方必须在 1 周内提供兼容方案，否则甲方有权要求更换技术栈。

#### 6. 验收与上线筹备清单
- [ ] **安全验收**：第三方渗透测试报告（零高危），代码审计报告无遗留问题。
- [ ] **性能验收**：核心交易响应时间 ≤ 100ms，峰值并发达到合同约定值的 120%。
- [ ] **灾备验收**：灾备切换演练成功，RPO/RTO 满足监管要求。
- [ ] **文档移交**：系统架构文档、接口规范、安全审计报告、运维手册、应急预案完整移交。
