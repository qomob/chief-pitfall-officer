# 七维拆解输出模板

本文档定义了每个维度的标准输出格式和填写规范。执行拆解时严格按此模板输出。

---

## 正式工作目标声明模板

```markdown
### 正式工作目标声明

*   **项目名称**：[简洁有力的项目名，含行业属性]
*   **核心目标**：[一句话概括，必须包含：约束条件 + 预期成果 + 量化指标]
*   **关键约束**：[预算上限 / 时间截止 / 范围排除项，逐条列出]
*   **成功标准**：[2-3 个可量化的验收指标]
```

**填写规范**：
- 项目名称应包含行业属性和项目类型（如"XX行业XX项目"）
- 核心目标不超过两句话
- 关键约束必须是硬性限制，不含弹性表述
- 成功标准必须能量化验证

---

## 维度一：工作内容拆解（WBS）

```
[项目总任务：XX]
├── 阶段一：[阶段名]（第X-Y天）
│    ├── 子任务1.1：[动词开头的具体动作]
│    ├── 子任务1.2：[动词开头的具体动作]
│    └── 子任务1.3：[动词开头的具体动作]
├── 阶段二：[阶段名]（第X-Y天）
│    ├── 子任务2.1：[具体动作]
│    └── 子任务2.2：[具体动作]
├── 阶段三：[阶段名]（第X-Y天）
│    └── ...
└── 阶段N：[阶段名]（第X-Y天）
     └── ...
```

**填写规范**：
- 阶段数量：3-4 个，不超过 5 个
- 每阶段子任务：2-5 个
- 子任务必须以动词开头（如"设计""安装""审核""发布"）
- 时间标注格式统一为"第X-Y天"或"Day X"
- 有并行阶段的需在括号内标注"（与阶段X并行）"

## 维度三：岗位职能匹配与风险预警

```markdown
*   **[角色名]（[人员来源：内部/外包/兼任]）**
    *   *职能*：[2-3 句具体职责描述]
    *   *协作*：[主导/配合/审批/咨询] — [与哪些角色如何配合]
    *   *风险预警*：[该角色常见的推诿/风险场景]
    *   *防范方案*：[具体的甲方防范/审计措施]
```

**填写规范**：
- 每个角色标注人员来源（内部/外包/兼任）
- 职能描述具体到可交付成果
- 协作关系用"主导/配合/审批/咨询"标注
- 必须包含针对该角色的风险预警和防范方案

## 维度四：流程动线与环境核查

```
[前置任务] ──(依赖)──> [后续任务] ──> [下一环节]
                                     │
                                     ▼
                              ★ [关键决策点：决策内容]
                                     │
                                     ▼
[并行任务A] ←───────→ [并行任务B]
```

#### 环境核查清单 (Checklist)
- [ ] **技术环境**：[针对该流程的具体技术要求]
- [ ] **政策环境**：[相关的法规/审批节点]
- [ ] **供应链环境**：[物料/三方服务依赖]
- [ ] **市场环境**：[竞品/外部动态监控]

**标注规范**：
- `──(依赖)──>` 表示必须先完成后才能开始
- `★` 标记关键决策点（Gate Review）
- 环境核查清单需针对本项目流程中的关键节点进行具体化

```markdown
*   **[评估维度]（[风险等级：高/中/低]）**：

    - *对策*：[具体应对策略]

*   **可行性结论**：[基本可行 / 有条件可行 / 需重大调整]

```

**填写规范**：
- 必须覆盖时间、资金、运营三个维度

- 对策必须可执行，不能是"加强管理"等空话
- 最终结论只能是三选一，不允许模棱两可

```markdown
### 方案选型

| 方案 | 核心思路 | 优势 | 劣势 | 适用条件 |
```

## HTML 杂志风转换与填充规范

生成专业级 HTML 报告时，需将 Markdown 内容映射至 `<skill-base>/assets/report-template.html` 中的高级占位符。

### 占位符映射表 (Magazine v2)

| 占位符 | 处理要求 (Magazine Style) |
| :--- | :--- |
| `{{PROJECT_DURATION}}` | 提取 WBS 中的总天数 (e.g., "45 Days") |
| `{{PROJECT_RISK}}` | 提取可行性评估中的最高风险等级 (e.g., "Low Risk") |
| `{{SUCCESS_TAGS}}` | 将成功标准转换为胶囊标签: `<span class="px-3 py-1 bg-white/10 rounded-full text-xs">...</span>` |
| `{{SUCCESS_CRITERIA_LIST}}` | 转换为带 Checkmark 图标的垂直列表 |
| `{{CONSTRAINTS_CARDS}}` | 转换为 3 列图标卡片，每张卡片含标题与描述 |
| `{{SELECTION_TABLE_MAGAZINE}}` | 转换为精美的杂志风表格，推荐方案行需加 `bg-indigo-600/5` 强调色 |
| `{{WBS_MAGAZINE_LAYOUT}}` | 转换为垂直时间轴或阶段卡片，支持 Alpine.js 折叠 |
| `{{SKILLS_GRID_MAGAZINE}}` | 转换为 2 列或 3 列的网格卡片，突出工具/平台名称 |
| `{{ROLES_CARDS_MAGAZINE}}` | 转换为带头像占位符/角色图标的视觉卡片 |
| `{{BUDGET_VISUAL_MAGAZINE}}` | 转换为可视化比例条（Progress Bars）与分类明细 |

### 核心美学原则 (Fused from Guizang PPT)

1.  **字体分工**：
    -   大标题/数字：使用 `Playfair Display` 或 `Noto Serif SC` (900/Bold)。
    -   正文：使用 `Noto Sans SC` (Light/Regular)，行高保持在 1.6-1.8。
    -   标签/元数据：使用 `IBM Plex Mono`，大写并加宽字符间距 (`tracking-widest`)。
2.  **布局节奏**：
    -   **Hero Slide**：极简大字报风格，WebGL 氛围背景。
    -   **Bento Slide**：利用 Bento Grid 将异构信息（目标+成功标准+约束）整合在同一屏内。
3.  **色彩控制**：
    -   背景色使用 `--paper`，文字使用 `--ink`。
    -   强调色仅用于 kicker、图标、强调线及推荐方案，保持画面克制。
